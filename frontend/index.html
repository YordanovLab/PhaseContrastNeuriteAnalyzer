How to use this project right now

  1. Use the functional app for running the backend pipeline steps.
  2. Use this help page for orientation, explanations, and architecture overview.
  3. Confirm the shared settings first.
  4. If you use WSL for Fiji, make sure GUI support is enabled because GUI-capable Fiji mode is preferred for reproducible output.
  5. Check whether a saved validated cutoff profile already exists.
  6. Choose either reuse of that saved profile or a fresh validation cycle.

Reference Library

Embedded backend guidance

These sections restate the important backend text in a frontend-readable way, so the user can stay in one place.

Prepare image inputs

The first step standardizes image identity before any analysis starts. The renaming script creates a PRE_renamed/ folder, keeps the nested directory hierarchy, and creates uniquely named hard-linked copies derived from the folder path. The image list script then scans that folder and writes absolute paths into OUT_img_input1.txt so later stages can batch-process the same image set reliably.

Settings and required functionality

The project now uses a shared settings file, config/pipeline_settings.env, as the first layer of control. The most important early continuity file is OUT_img_input1.txt, because it freezes the exact list of images that Fiji preprocessing will use. The relevant entry scripts now point to the shared settings file for paths, output folder names, the Fiji binary, the ilastik binary, and the ilastik project.

Saved validation cutoff decision

The first important branch in the workflow is whether a trusted validated cutoff profile already exists. If not, the correct logic is: Validation with Test Images, then Generalization with New Images, then output plotting for Visualization and Interpretation. If a saved validated cutoff profile already exists, the user should be allowed to either run a fresh validation again or reuse one saved profile chosen by date or folder name. In the reuse case, the workflow should continue directly with generalization and then visualization and interpretation.

Fiji preprocessing

The Fiji preprocessing stage converts each raw RGB image into a single standardized intensity image. The ImageJ macro opens each file, splits the red, green, and blue channels, computes a pixel-wise average of the red and green channels, saves the derived image as an _RGavg.tif output, and closes open images before moving to the next file so memory use remains stable.

Ilastik segmentation

The ilastik stage applies a trained batch model to the preprocessed images and writes segmentation masks into OUT_segmentation/. After prediction, each mask is renormalized so downstream scripts can rely on stable label values. This is the bridge between raw image intensity and later quantitative neurite measurements.

Mask quantification

The mask quantification macro reads the segmentation outputs, builds a 256-bin histogram for each image, and computes class-wise area fractions using the label mapping confirmed in the functional app. The current default mapping is value 1 for cell bodies, value 3 for neurites, and value 255 for background, but the app includes a dedicated segmentation-label check before validation grouping so exchanged labels can be fixed. It writes a single CSV table with image name, cell body percentage, neurite percentage, background percentage, and total area in pixels.

Manual validation loop

The validation loop first groups images into biologically useful quadrants, then asks the user to manually separate good examples from noise. The grouping script merges metadata with mask percentages, derives confluence, removes outliers, and relaxes high and low quantile cutoffs until enough candidate images are found in each quadrant. After manual sorting, the inspection script writes those human labels back into the master table.

ROI, optimization, and biological analysis

After validation, the pipeline extracts ROIs and optional skeleton summaries, packages the manually curated images into an optimization-ready R object, and searches ROI threshold rules that best capture neurite biology while resisting confluence artefacts and noise. The later R scripts then generalize the signal rule and carry it into image-level biological analysis, dimensionality reduction, and final reporting plots.

Visualization and Interpretation Module

This module starts only after earlier scripts have already created ROI, ROI+skeleton, optimization, or generalized biological outputs. The most useful upstream builders are 1.analyze-combine-ROIs-skeletons.R and 2.analyze-combine-ROIs-skeletons.R, which prepare analysis objects later scripts can visualize. The preferred visualization layer is 3.analyze-visualize-check.R for representative ROI views and 3.analysis-check-visualize-readiness-on-test-images2.R for richer class-comparison and PCA-style interpretation. The main production interpretation and reporting endpoints are 3.1.analyzed-generalized-skeleton-data-PCA+ALMOST_WORKING.R and 4.1.Skeleton-cutoff-ploting-improved-polished+BG.R.

Frontend principles used by the app

  • One vertical workflow, not a crowded sidebar-first layout.
  • One clear card per stage with inputs, outputs, and launch action.
  • Manual steps should be visibly different from automated steps.
  • Every stage should show where outputs were written.
  • The interface should be honest about which engine runs each step.

Current functional app tab structure

The Shiny app is now organized as a practical workbench rather than only a script launcher. The static help page should stay synchronized with this structure.

  1. 1. Configuration: choose project paths, metadata CSV, input directory, output workspace, Fiji, ilastik, and the trained ilastik project.
  2. 2. Validate New / Use Prior Cutoffs Model: either validate new cutoffs with test images or reuse a saved cutoff profile.
  3. 3. Analyze Test Images: apply one or more selected cutoff profiles to all production ROI folders and export the generalized image-level feature table.
  4. 4. Visualization and Interpretation: inspect training-image roles, PCA/separation, feature screening, grouped plots, covariate correlation, and ANCOVA.
  5. 5. Reproducible Report: generate a downloadable HTML report plus an archive of settings, scripts, logs, selected profiles, key outputs, and recreation notes.

Training the ilastik pixel classifier

The ilastik model should be trained on the same image type that the production pipeline will segment. In this workflow, that means using Fiji-preprocessed OUT_clear_bcgnd/*_RGavg.tif images, not arbitrary raw images from a different preprocessing state.

  1. Run setup through Fiji preprocessing so the training images have the same background correction / RG-average transformation used later.
  2. Create an ilastik Pixel Classification project.
  3. Add representative preprocessed images spanning low/high confluence, low/high neurites, different treatments, and imaging noise.
  4. Create three labels: background, cell bodies, and neurites.
  5. Paint examples for empty background and illumination artefacts; soma/cell-body regions and dense cell bodies; thin neurites, branches, and long interrupted neurites.
  6. Save the trained .ilp project outside git and point ILASTIK_PROJECT_FILE to it in config/pipeline_settings.env.
  7. After segmentation, use the app label-check tab to confirm that downstream labels map to cell bodies = 1, neurites = 3, and background = 255, or remap them if ilastik produced a different order.

GitHub repository readiness

The public GitHub repository should stay small and code-focused. Keep generated outputs, raw images, ROI folders, run logs, local settings, and trained ilastik project files out of git. Use config/pipeline_settings.example.env as the committed template and create config/pipeline_settings.env locally.

  • Use .gitignore to exclude pipeline_inputs/, pipeline_outputs/, cache/, TIFF files, ROI files, and trained models.
  • Upload ~5 GB raw microscopy datasets to Zenodo, Figshare, or OSF; Zenodo is recommended because it provides a persistent DOI and supports datasets of this size.
  • Put dataset and model DOI/download links in README.md after upload.
  • Use tab 5 in the app to produce a reproducibility archive for each finalized analysis.
